# Safety Protocol

この文書は、安全にゲーム制作を続けるための手順です。

## 目的

このプロジェクトでは、ユーザーが主にアートと企画を担当し、Codexが主にプログラムを担当します。コードが壊れたときにユーザーが直接直せない前提なので、作業は常に復旧しやすく進めます。

## 基本方針

1. 小さく作る
2. 動く状態を保つ
3. 変更理由を残す
4. GitHubへ定期的に退避する
5. 新しいCodexセッションでも再開しやすくする

## 作業前チェック

- `git status` で未コミット変更を確認する。
- 触る予定のファイルを先に読む。
- 既存の構造に合う実装方法を選ぶ。
- 不明点がゲーム仕様に影響する場合は、仮定を明記するかユーザーに確認する。

## 作業中チェック

- ひとつの機能を小さな単位に分ける。
- 大きなシーンやスクリプトを作りすぎない。
- プレイヤー、武器、弾、敵、UIなどの責任を分ける。
- データ化できるものは、将来的に `Resource` に逃がせる形で作る。

`Resource` とは、Godotで使う「データの入れ物」です。たとえば武器の攻撃力、弾数、リロード時間をコードに直接書かず、武器データとして保存できます。

## 作業後チェック

- `git diff` で自分の変更を確認する。
- Godotエディタまたはヘッドレス起動で、最低限プロジェクトが読めるか確認する。
- 可能なら該当機能を手動テストする。
- `docs/HANDOFF.md` に進捗、次の作業、注意点を書く。
- 意味のある単位でコミットする。

## Git運用

- `main`: 安定版。原則として壊れた状態を置かない。
- `feature/...`: 新機能用ブランチ。
- `fix/...`: バグ修正用ブランチ。
- `backup/...`: 大きな変更前の退避用ブランチ。

コミットメッセージは短く具体的にします。

例:

```text
Add project safety documentation
Implement basic player movement
Fix reload ammo calculation
```

## バックアップ方針

- 作業の節目でGitHubへpushする。
- プレイ可能な節目ではタグを作る。
- 大きなリファクタ前には退避ブランチを作る。
- アート素材が大きくなったらGit LFSを検討する。

`Git LFS` とは、大きな画像や音声ファイルをGitで扱いやすくする仕組みです。必要になるまでは導入しません。
